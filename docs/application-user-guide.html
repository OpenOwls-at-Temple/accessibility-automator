Accessibility Automator · User Guide

Remediate a PowerPoint & read your accessibility scores

A short, practical guide to running the app and seeing the before / after improvement on your lecture files.

Section 01

Two ways to run it

Pick whichever fits what you want to do. Both produce the same remediated file and the same accessibility scores.

Fastest · one command

Command line (CLI)

Point it at a .pptx (or .pdf) and it prints the before/after scores and writes the fixed file plus a report. Best for a quick test.

Full experience · in your browser

Web app

Sign in, upload files into a course group, click Fix, watch progress, and view the report — the way faculty will actually use it.

One-time setup. The project uses a Python virtual environment at .venv (Python 3.11+). All commands below assume you are in the project root folder accessibility-automator.
Section 02

Quick test with the command line

From the project root, run the fix command on your PowerPoint:

# from the project root folder uv run python -m backend.cli fix /path/to/your_deck.pptx

You'll see output like this:

Remediated: your_deck.pptx -> your_deck_a11y.pptx Score before: 44 Score after (checker): 100 Score truly remediated: 75 Genuine fixes: 3 Placeholders (follow-up): 2 Needs manual fix: 0 Reports: your_deck_a11y.pptx.report.json, your_deck_a11y.pptx.report.html

Three files are written next to your input:

your_deck_a11y.pptxThe remediated PowerPoint — open it in PowerPoint, or re-upload it to Canvas.
…_a11y.pptx.report.htmlA readable before/after report. Open this in any browser.
…_a11y.pptx.report.jsonThe same report as machine-readable data.
Your original is never touched. The app always writes a new file with an _a11y suffix and leaves your source file exactly as it was.
Section 03

Full experience in the web app

The web app needs two pieces running: the backend (API) and the frontend (the page you click around in). Start each in its own terminal.

Terminal 1 — start the backend

# from the project root cp .env.example .env # first time only uv run alembic upgrade head # first time only — create the users table uv run python -m backend.app.seed --admin you@temple.edu # first time only — make yourself admin uv run uvicorn backend.app.main:app --reload # now serving the API at http://localhost:8000

Terminal 2 — start the frontend

cd frontend cp .env.example .env # first time only npm install # first time only npm run dev # now serving the app at http://localhost:5173

Then, in your browser at http://localhost:5173

  1. Sign in using the Local dev login box with your registered email — e.g. the you@temple.edu you seeded as admin above. (In production you'd click Sign in with Google; access is invite-only, so an admin adds each Temple account first.)
  2. Click Upload, type a group name (usually your course code, e.g. CIS4526), and choose your .pptx file.
  3. Click the Fix button on the group. The button shows live progress while it works.
  4. When it finishes, the file row shows three scores: Before → After → Truly fixed.
  5. Click report for the full breakdown, or remediated to download the fixed _a11y.pptx.
Heads up: the backend must be running for the web app to work, and its allowed origins already include http://localhost:5173 (the default). Stop either server with Ctrl + C in its terminal.
Section 04

Understanding the before & after scores

Every file gets a score out of 100, the same idea Canvas's accessibility checker (YuJa Panorama) uses — higher is better, weighted so more serious issues cost more. The app shows three numbers:

44
Before
Your original file, as uploaded.
100
After (checker)
What Canvas will likely report. Placeholders count as passing.
75
Truly remediated
The honest number — placeholders excluded.

Why two "after" numbers? When the app can't be sure of a genuine fix (for example, it couldn't confidently describe an image), it inserts a placeholder so the file still passes the automated checker — and flags it for you. The gap between "After (checker)" and "Truly remediated" is exactly the work that still needs a human. The report lists those items so you know what to review.

Genuinely fixedReal fixes the app made on its own (titles, table headers, language, AI alt text, …).
PlaceholdersPassed the checker with a stand-in value — needs your review. You can acknowledge each in the report.
Needs manual fixIssues the app detected but can't safely change automatically (e.g. color contrast).
The real confirmation is Canvas. The app's score is a close approximation; the official score appears when you re-upload the _a11y file to Canvas. "Passes the checker" is not the same as "perfectly accessible" — that's why the honest second number exists.
Section 05

Getting real AI alt text (optional)

Out of the box, the app still runs without any AI configured — it writes placeholder alt text and slide titles so files pass the checker, and flags them for review. To get genuine, AI-generated descriptions, set three values before you start the CLI or the backend:

export LLM_API_KEY=sk-your-key-here export LLM_BASE_URL=https://api.openai.com/v1 # or any OpenAI-compatible endpoint export LLM_MODEL=gpt-4o-mini # a vision-capable model

With these set, images get real descriptions and untitled slides get real titles — which raises the "Truly remediated" score, not just the checker score.

Privacy: only the image and a little surrounding slide text are sent to the model, and only ever from the server. No student data is involved — the app handles faculty course materials only.
Section 06

Quick troubleshooting

Unsupported filePhase 1 accepts .pptx and .pdf only. Other types are rejected with a message.
PDF score stays lowExpected for untagged/scanned PDFs — the app fixes the title and language but flags the deeper structural work (and OCR) for follow-up. PowerPoint files reach a much higher score.
Frontend can't reach APIMake sure the backend is running on http://localhost:8000 and that frontend/.env points to it.
Alt text looks genericThat's a placeholder — set the LLM_* values (Section 05) for real descriptions.
