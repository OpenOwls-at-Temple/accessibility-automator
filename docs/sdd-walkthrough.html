Spec-Driven Development in Action

How a single planning conversation turned a fuzzy idea into a complete, code-ready specification — before a single line of code was written.

Case study: the “Accessibility Automator” project
The Idea

From a vague idea to a plan

A faculty member started with a real problem: at Temple, lecture files uploaded to Canvas are scored for accessibility by a tool called YuJa Panorama, and fixing every flagged issue by hand is slow and tedious. The idea was simple to say but fuzzy to build: “an app that automatically makes my PowerPoint and PDF files accessible.”

Instead of jumping straight to code, they followed the OpenOwls SDD process: talk the idea through, make the decisions explicit, and write them into the ai_specs/ files. Only then does the project move to Claude Code for implementation. This page replays that conversation so you can see how the specs got built — and why each step matters.

Fuzzy idea → Clarifying questions → Decisions + research → ai_specs/ files → Claude Code builds it
The Walkthrough

How the spec was built, step by step

Each step below is a real moment from the planning session. The green box names the SDD principle it demonstrates.

  1. 1

    Pin down what the app actually is

    Before anything else: who is this for, and what does it do? The discussion settled on a clear one-liner — a standalone web app that remediates faculty PPTX/PDF files against WCAG so they score near-100 in Canvas.

    SDD principle: Start with overview.md — the problem, the user, and the goal — before touching features or tech.
  2. 2

    Ask the questions that change the design

    Rather than assuming, the big forks were raised out loud: Should the app integrate with Canvas, or just produce fixed files? Web app or desktop? Which file types first?

    Decision: A standalone tool (Option A) for Phase 1; the Canvas integration is parked for Phase 3.
    SDD principle: Resolve ambiguity before writing specs. A good question now saves a rewrite later.
  3. 3

    Scope a real MVP with phases

    The temptation is to build everything at once. Instead, features were sorted into three phases — and the sponsor was reminded when Phase 1 was getting large.

    Phase 1: PPTX + PDF remediation, sign-in, file workspace, the checker, AI alt-text, and the report.
    Phase 2: Word, video, batch, human review.
    Phase 3: Canvas + Panorama integration.
    SDD principle: features.md forces a real MVP. A good Phase 1 is one workflow, working end-to-end.
  4. 4

    Catch conflicts between spec files

    When the sponsor added Microsoft SSO and per-user folders, that contradicted a non-goal already written in overview.md (“no authentication, stateless, no storage”). The conflict was flagged and resolved — auth moved in-scope, behind a swappable interface so it wouldn’t block the build.

    SDD principle: The specs must stay consistent with each other. Flag conflicts and fix them — never code over them.
  5. 5

    Research the domain before designing

    The team paused to look things up: the standard is WCAG 2.1/2.2 AA; Temple’s checker is YuJa Panorama; scoring is a weighted average by severity. A key finding — Panorama often misses PDF issues — became a documented constraint.

    SDD principle: domain-knowledge.md captures the facts the code depends on, so the AI doesn’t guess.
  6. 6

    Design the architecture and justify the decisions

    The backend was designed around a decoupled engine with per-format rules and a shared scorer, plus per-user storage and a clear set of API endpoints. Each major choice was written down with its reason, not just the choice itself.

    SDD principle: architecture-planning.md is where you make and justify design decisions before writing code.
  7. 7

    Treat the LLM as a first-class component

    The AI’s job was defined precisely: caption images and name untitled slides — returning a confidence score so low-confidence output becomes a flagged placeholder instead of a fake fix. It sits behind an OpenAI-compatible interface so the model can be swapped.

    SDD principle: llm-integration.md makes prompts, fallbacks, and evaluation explicit — not an afterthought.
  8. 8

    Set the rules, then record the state

    Finally, conventions.md (naming, testing, the one-way engine dependency) and deployment.md (Render, env vars, SSO setup) were filled in — and progress.md was updated so the next session starts informed.

    SDD principle: Update progress.md every session — Claude Code has no memory between sessions.
A Design Value

An honest tool, by design

One theme ran through the whole conversation: a placeholder like “image” can make the checker say 100%, but it doesn’t actually help a student using a screen reader. So the spec bakes in honesty — the report shows two numbers:

✅

Checker-passing score

What Canvas/Panorama will report — placeholders count as passing.

🎯

Truly-remediated score

Excludes placeholders. The gap is the human follow-up list.

This is exactly the kind of value judgment that belongs in the specs, where everyone can see and agree to it — long before it’s buried in code.

The Deliverable

The seven spec files we produced

Every decision above landed in a file. This is the complete, code-ready spec set handed to Claude Code:

FileWhat it captures for this project
overview.mdThe app, its users, goals/non-goals, and the React + FastAPI stackdone
features.mdPhase 1/2/3 features with acceptance criteria and the WCAG check listdone
domain-knowledge.mdWCAG, YuJa Panorama, scoring, and the “checker vs. truly accessible” constraintdone
architecture-planning.mdThe decoupled engine, per-format rules (P1–P13 / D1–D21), API, storage, configdone
llm-integration.mdAlt-text/title prompts, confidence→placeholder fallback, evaluationdone
conventions.mdNaming, testing, git, and the one-way engine dependency ruledone
deployment.mdRender hosting, env vars, Tesseract, and Azure SSO setupdone

Plus the two root files: CLAUDE.md (tells Claude Code what to read, in order) and progress.md (the living session log).

For Students

What to take away

  1. Specs before code. The hardest decisions — scope, architecture, the LLM’s role — were all made in plain language first, when changing course was cheap.
  2. Ask before you assume. Each clarifying question removed a wrong path the project could have gone down.
  3. Phase ruthlessly. Everything that wasn’t essential to one working workflow got pushed to Phase 2 or 3.
  4. Keep the specs consistent. When two files disagreed, that was a bug to fix in the docs — not something to discover later in the code.
  5. Ground it in facts. Real research (WCAG, Panorama) turned guesses into documented domain knowledge.

The payoff: when this project opens in Claude Code, the AI already knows what to build, how to build it, and what “done” looks like — because the thinking is already written down.
